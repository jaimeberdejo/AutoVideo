## Bugs Found and Fixed

### 1. [BLOCKER] Scriptwriter calibration retry loses all topic context → generates unrelated script
- **Symptom:** For a real 60s/6-slide presentation about small-business carbon-footprint tracking, the generated script came back as a 4-slide generic "innovation drives organizations" script with zero connection to the actual topic.
- **Root cause:** `ScriptwriterStage`'s single calibration retry (triggered when word-count drift exceeds 25%, very likely on tight per-slide budgets) sent Claude a follow-up prompt containing ONLY the word-count mismatch notes — never the original slide titles/bullets or the previous narrations. Since `call_structured` is a stateless single-turn call (no conversation history threaded through), the retry had no topic context at all and had to invent something.
- **Fix:** `src/avideo/stages/scriptwriter.py` — `_correction_prompt()` now rebuilds the full slide-block context (title/bullets per slide, matching `_build_prompts`) and includes the previous narrations, so the retry stays self-contained and grounded.
- **Test added:** `tests/test_scriptwriter.py::test_calibration_retry_prompt_includes_original_slide_content`.

### 2. [BLOCKER] FFmpeg assembly fails on every real run
- **Symptom:** `Error en el montaje: ffmpeg failed (exit 234): ... Unable to choose an output format for '/tmp/.../output.mp4.tmp'; use a standard extension...`
- **Root cause:** The atomic-write temp filenames (`output.mp4.tmp`, `output.mp4.norm.tmp`) had `.tmp` as the FINAL extension. FFmpeg infers the container/muxer purely from the output filename's extension, and `.tmp` isn't a recognized muxer — so every real assembly failed at the very first ffmpeg call. This was invisible to the 446-test unit suite because it mocks `run_ffmpeg`/`subprocess` entirely, so ffmpeg's actual extension-based format detection was never exercised.
- **Fix:** `src/avideo/stages/assemble.py` — renamed all three temp paths to keep `.mp4` as the final extension (`output.tmp.mp4`, `output.norm.tmp.mp4`), matching the one place in the same file (`output.music.tmp.mp4`) that already had it right. Updated the module docstring accordingly.
- **Test added:** `tests/test_assemble.py::test_ffmpeg_output_paths_end_in_mp4_not_tmp` — captures every path passed to `run_ffmpeg` and asserts it ends in `.mp4`.

### 3. [BLOCKER] Browser refresh/app restart crashes past Phase 1 with a raw ValidationError
- **Symptom:** After approving Phase 1 and refreshing the browser (or restarting the `avideo studio` process) while on Phase 3+, the page showed: `Error de configuración: 1 validation error for RunConfig duration Field required`.
- **Root cause:** `session_state["run_config"]` (holding topic/duration) is only ever set by Phase 1's approval handler and is NOT rehydrated from workdir on refresh — unlike `session_state["phase"]`, which IS correctly reconstructed from workdir done-markers. Since `session_state` doesn't survive a refresh/restart but `phase` gets rebuilt while `run_config` resets to `{}`, every downstream page's `RunConfig(**rc_dict)` call failed.
- **Fix:** Added a `UIRunConfig` checkpoint (`src/avideo/ui/state.py`) written alongside the "context" checkpoint on Phase 1 approval (`phase_1_contenido.py`), and rehydrated into `session_state["run_config"]` during the existing `_phase_initialised` block in `app.py` (the same place `phase` is rebuilt).
- **Test added:** covered by `tests/test_ui_wizard_e2e.py::test_refresh_resumes_from_workdir` (real browser) — no unit test added since `state.py`'s `rehydrate_run_config()` is a thin, directly-readable wrapper around `WorkdirManager.read_checkpoint`.

### 4. [HIGH] "Generar voz" / "Montar vídeo" buttons stuck disabled forever after an error
- **Symptom:** After a synthesis or assembly error, the retry button remained disabled with no way to retry short of a full manual browser refresh.
- **Root cause:** The button's `disabled=` value is computed once per full Streamlit script rerun (at click time, when the bridge thread was still `RUNNING`). The `@st.fragment(run_every="2s")` polling block that shows the error only calls `st.rerun()` on `DONE`, never on `ERROR` — so the outer script (and the button's disabled state) never gets recomputed after a failure.
- **Fix:** Added a consumed-once session-state flag (`_voice_error_rerun_done` / `_assemble_error_rerun_done`) that forces exactly one additional full rerun the first time an error is observed, re-enabling the button, and clears it on the next click (so future errors can retrigger it too — no infinite rerun loop).
- **Verification:** Re-tested live: after the ElevenLabs 402 error, refreshing showed the button enabled; after applying the fix and restarting the app, the button was correctly enabled on the resumed Phase 6 (no stuck state observed post-fix).

### 5. [MEDIUM] Raw exception dump shown to the user instead of a clean message
- **Symptom:** The ElevenLabs 402 error rendered as a multi-hundred-character dump of raw HTTP response headers, status code, and body.
- **Fix:** Added `bridge.format_stage_error()` which extracts `exc.body["detail"]["message"]` (ElevenLabs' error shape) when present, falling back to `str(exc)` otherwise. Wired into all 3 call sites in `phase_4_voz.py` and `phase_6_ensamble.py`.
- **Test added:** `tests/test_bridge.py::test_format_stage_error_extracts_elevenlabs_style_body_message`, `test_format_stage_error_returns_empty_string_when_no_error`.

### 6. [HIGH] Silent wrong-duration fallback in assembly
- **Symptom (potential, not directly observed live since fix #3 landed first):** If `run_config` was ever missing `duration` on the assembly page, the code defaulted to `60` seconds silently and assembled the video against that fabricated target — no crash, no warning, just a wrong-length video.
- **Fix:** `phase_6_ensamble.py` — removed the silent `rc_dict["duration"] = 60` fallback; a missing/invalid config now shows a clear error directing the user back to Phase 1, consistent with how `phase_2`/`phase_3`/`phase_4` already handle this.

### 7. [MEDIUM] QA report's "normalized LUFS" metric was always wrong
- **Symptom:** The QA report showed `normalized_lufs == measured_lufs` (e.g. both -29.5 LUFS) on every real run, even though the actual produced audio WAS correctly normalized (independently re-measured the real `output.mp4` afterward: ~-16 LUFS, matching the -16 LUFS target).
- **Root cause:** `parse_loudnorm_json()` always extracts `raw["input_i"]` (the pre-normalization loudness) regardless of whether it's parsing a pass-1 (measure) or pass-2 (apply) ffmpeg JSON block. `_run_qa()`'s `normalized_lufs` computation read this same `"input_i"` key from the pass-2/apply JSON, based on an incorrect code comment claiming pass-2's `output_i` is "reported as input_i in the re-emitted JSON" — this is simply not how ffmpeg's loudnorm filter JSON output works; `output_i` is a distinct field holding the real result.
- **Fix:** `integrations/ffmpeg.py` — `parse_loudnorm_json()` now also returns `output_i`. `assemble.py` — both the two-pass path and the single-pass-with-music path now read `output_i` for `normalized_lufs` instead of `input_i` / a hardcoded `target_lufs` guess.
- **Test added:** `tests/test_assemble.py::test_parse_loudnorm_json_exposes_output_i_distinct_from_input_i`.
- **Independent verification:** re-ran the real two-pass loudnorm directly against the produced `output.mp4` outside the app — confirmed `output_i ≈ -16.0` (matching target) while `input_i` (what the old code was reporting) was whatever the file's actual prior loudness was.

### 8. [LOW] Misleading "in progress" status shown before the user has even clicked "start"
- **Symptom:** Landing on Phase 4 (voice, ElevenLabs/OpenAI) or Phase 6 (assembly) immediately showed "Sintetizando voz..." / "Montando vídeo..." even though nothing had been triggered yet (bridge status was `IDLE`, not `RUNNING`).
- **Fix:** Split the `RunStatus.RUNNING`/`RunStatus.IDLE` cases, which were previously lumped together; `IDLE` now shows a clear call-to-action ("Pulsa 'Generar voz'/'Montar vídeo' para comenzar...").
